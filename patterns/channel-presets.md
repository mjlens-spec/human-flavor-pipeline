# 载体、受众、文体与声口

「飞书」「Notion」「公众号」只是载体,不能直接推出文体。调用时分四步判断:

1. **载体**:只决定 Markdown、富文本、引号、空格、链接等排版能力。
2. **受众**:团队内部、专业同行、客户决策者、公众读者等,决定术语解释深度。
3. **文体**:商务提案、技术说明、叙事特稿、品牌文案、公文法律等,决定结构与语言规则。
4. **声口**:直白、温和、犀利、技术、叙事、口播,决定节奏与锋芒。见 `voice-profiles.md`。

用户只给载体时,不得自动补出受众和文体;从正文推断仍不确定,就在报告里声明假设。

---

## 载体层(只管格式)

| 载体 | Markdown | 引号 / 标点 | 注意点 |
|------|----------|-------------|--------|
| 飞书文档 | 由客户端渲染 | 服从用户规范 | 定点写回,保护图片与评论 |
| Notion | 支持块与 Markdown 导入 | 服从用户规范 | 容器既可放内部笔记,也可放客户方案 |
| 微信公众号编辑器 | 不把源码标记直接交付 | 通常用中文全角标点 | 长文、通知、品牌稿都可能出现,需另判文体 |
| 纯文本 / 邮件 | 不假定支持 Markdown | 按收件人和组织规范 | 列表、链接需在纯文本里可读 |
| Markdown 仓库 | 支持 Markdown | 代码与正文分开 | `#`、`**`、代码围栏是合法格式,不算残留 |

破折号、分号和省略号没有统一的「真人基线为 0」。只在功能错配、密度异常且文体不支持时处理。

## 受众层

- **团队内部**:允许团队约定的英文词、缩写、项目代号和行业术语;准确优先于通俗。
- **专业同行**:保留标准术语,只解释可能有歧义的新词。
- **客户决策者**:术语可保留,但要让动作、责任、预算和口径可核对。
- **公众读者**:必要时解释术语,不默认把全部英文翻成中文。

技术英语的判断顺序:团队约定 > 行业惯例 > 受众理解成本 > 作者个人声口。`latency`、`throughput`、`context window` 等在工程团队文档里可以原样保留;面向非技术公众时再判断是否需要中文解释。

## 文体层

| 文体 | 保护项 | 重点清理 |
|------|--------|----------|
| 商务 / 职场 | 准确枚举、节点日历、KFS、ROI 口径 | 空承诺、无指标愿景、机械套话 |
| 技术 / 学术 | 术语、被动句、逻辑连接、正式枚举 | 无来源断言、重复解释、术语误用 |
| 叙事非虚构 / 特稿 | 铺垫、人物声口、合法破折号 | 空抒情、模板升华、事实漂移 |
| 口播 / 播客逐字稿 | 第一人称、口头转场、反问、适量软垫、可朗读的回环 | 书面路标、机械排比、无来源亲历、伪造口头禅 |
| 品牌广告 / 文案 | 短句、排比、留白、中英混排 | 商务僵尸词、模板 CTA、空洞拔高 |
| 公文 / 法律 | 规范术语、层级编号、固定结构 | 事实错误、口径冲突、非规范表达 |
| 社交 / 口语 | 语气词、emoji、圈内说法 | 密集模板、过时套话、助手尾巴 |

## 乙方腔的好坏边界(客户方案 / 提案专用)

**乙方 deck 结构不等于 AI 味。** 以下内容有明确业务功能时保留:

- 战略角色定义、KFS / 媒介分配逻辑、节点日历
- 人群分层、预算 / ROI 区间、责任与交付边界
- 「理由有三」「四个动作」等承载实质内容的枚举
- 需求拆解、项目落地、流量承接、转化链路等准确业务术语

需要清理的是删掉后不损失可执行 / 可验证信息的空话,例如没有时间、范围和指标的「全面提升品牌影响力」。

## 与用户规范的仲裁

事实与逐字引语 > 用户明确偏好 > 受众 / 文体要求 > 载体格式 > 通用规则。

- 中英空格、引号和破折号按用户明确规范执行。
- 用户没有给规范时,按目标载体与文体选择,不要从载体名称反推固定声口。
