# 设计说明

这条管线不是凭空设计的,是综合了 GitHub 上口碑较好的一批去 AI 味 / humanizer 项目的共识做法,再结合中文专业写作(营销策划、客户方案、长报道特稿、飞书 / Notion / 公众号交付)的真实工作流重构出来的。

---

## 为什么是六阶段,而不是一个大 prompt

调研到的多数低质工具是「一个大 prompt 边读边改」,问题是:检测和改写混在一起,改坏了事实也发现不了;且无法对单一环节调参。口碑较好的项目(humanizer-skill 的 4-pass、shuorenhua 的双遍审读、humanizer-de 的 5-pass)都把流程拆成可独立调参的阶段。本管线据此拆成六段,关键是**检测与改写分离**、**收尾独立校验**。

## 关键设计决策

1. **先检测后改写(阶段 1 只读)** —— 借鉴 humanizer-skill 的 0–100 AI-tell score 作首尾两道闸,让「改前 → 改后」的落差成为可量化交付。
2. **先结构后词汇(阶段 2 顺序)** —— 结构性 AI 信号(三段排比、烘托 / 感召、机械列表)比词汇信号强,人类审稿一眼能看出。只换词、不动结构的稿子,即使骗过检测器也还是「读着像 AI」。
3. **保护区 ＋ 事实保真闸(阶段 0 ＋ 4)** —— 多 pass 高温改写的最大风险是篡改数字 / 实体。humanize-text 的回译流水线靠锁定信息保住 100% 关键信息,本管线用显式保护区清单 ＋ 收尾逐项核对实现同样目的。
4. **留毛边 / 反过度净化(阶段 3 ＋ 4)** —— 最激进的 humanizer 会把人味连同 AI 味一起削平,产出另一种无菌腔。强制保留至少一处真实毛边,但不许伪造。
5. **有界迭代,不无限催降** —— 中文圈普遍提醒「继续降到 10% 以下」式催促效果递减且过度净化。本管线改用强度档位 ＋ 最多 N 遍回改。
6. **渠道感知** —— 飞书内部 ≠ Notion 对外提案 ≠ 公众号特稿 ≠ 小红书。同一套词典按渠道门控才不会误伤。
7. **独立校验子 agent** —— 高风险场景派独立 agent 做对抗式复核,借的是 diff 式独立评审与多模型交叉验证的思路,避免主流程自评自己。

## 参考与致谢(同类项目)

设计共识来自对以下项目的调研,在此致谢:

- humanizer-skill(Aboudjem)—— 4-pass、43 patterns、0–100 score、voice profiles
- shuorenhua(说人话)—— 中文三级词典、双遍审读、强度分档
- Humanizer-zh(op7418)—— 24 类 AI 模式中文移植、质量评分矩阵
- humanize-text —— 回译流水线、关键信息保真
- StealthHumanizer —— 多模型交叉验证、迭代收敛
- humanizer-de —— 确定性 linting、多 pass 工作流

本项目对它们不是照搬,而是取其阶段化、分级、保真、留人味的共识,针对中文专业交付场景做重构。

## 不做什么

- 不以过 AI 检测器为目标(见 `scoring.md` 末)
- 不从零生成内容
- 不处理英文正文与繁体中文
- 不替代人的观察、采访、判断
